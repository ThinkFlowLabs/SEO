02 · The Five Critical Findings

What's actually
holding the site back.

Each of these was found by inspecting the live site or the search data, not by theory. Each has a fix that can ship inside 2 weeks. Cumulatively, they explain why a strong product is producing a weak SEO outcome.

· · ·
1
CRITICAL · TECHNICAL

The blog's canonical tag is sending equity to the wrong page.

Impact: 44,845 impressions on a single post → 39 clicks. 0.09% CTR. Repeats across every blog post.

Every blog post we inspected (/blog/7-days-anti-inflammatory-meal-plan, /blog/how-much-is-a-personal-chef-less-than-you-think) declares its canonical URL as https://chefpost.com/blog/, i.e. the index page, not itself. Google has been consolidating 16 months of links, impressions and behavioral signals into the index. That's why the anti‑inflammatory post still ranks well but produces almost zero clicks: the SERP snippet is sometimes pulled from the wrong URL, and the post's own equity is being attributed elsewhere.

Fix: Set <link rel="canonical" href="{currentURL}"> per post (self‑canonical). Same fix on /become-a-personal-chef (currently canonicals to /career/) and on every /chef/* profile (currently no canonical). Eta to ship: 1 dev day. Expected lift: 10–30× CTR on the affected URLs with no ranking change required.
2
CRITICAL · INDEXING

Five subdomains are indexing the same content as four different sites.

Impact: link equity, crawl budget and click data all fragmented.

Google has indexed Chefpost across at least five hostnames, every chef page potentially exists on 3 of them:

  • https://chefpost.com/ · canonical, 88,523 impressions
  • http://chefpost.com/ · non‑HTTPS still indexed (27,216 impressions, 424 clicks!)
  • https://www.chefpost.com/... · duplicates of chef pages
  • https://booking.chefpost.com/... · duplicates of chef pages
  • https://dev.chefpost.com/...,your development server is in Google's index. This is a security and SEO problem.
  • https://blog.chefpost.com/ · Substack blog on a separate subdomain, splitting authority from /blog/
Fix: Force 301 redirects to the canonical https://chefpost.com/{path} at the edge for every non‑canonical host. Immediately add HTTP basic auth to dev.chefpost.com + robots.txt: Disallow: / + X-Robots-Tag: noindex. Migrate the Substack blog into /blog/ with 301s. Eta: 2 dev days. Expected lift: link equity consolidation + Google sees a cleaner site architecture.
3
CRITICAL · SCHEMA

There is zero schema.org markup anywhere on the site.

Impact: ineligible for rich snippets, FAQ panels, recipe carousels, local pack, AI citations.

We inspected 12 different page templates. Not a single one ships JSON‑LD. The /faq page literally renders its Q&A as H2s and ships zero FAQPage schema, the lowest‑effort rich‑result win on the entire site. Chef profiles have no Person / Organization / LocalBusiness markup; blog posts have no BlogPosting / Recipe; the homepage has no Organization / WebSite / SearchAction; nothing for AggregateRating, even though chef reviews exist on the page.

The downstream cost is much larger than missing stars in the SERP: AI engines (ChatGPT, Perplexity, Gemini, Google AI Overviews) preferentially cite content with structured data. That's a big reason chefpost is currently invisible in LLM answers (see Section 09).

Fix: Ship a schema layer in 4 phases (detailed in Section 06). Phase 1 alone, Organization + FAQPage + BlogPosting, is a 2‑day implementation that immediately makes Chefpost eligible for FAQ rich results across ~30 pages.
4
CRITICAL · ON-PAGE

9 of 12 pages have no H1. The blog cost post has the title "| Chefpost".

Impact: Google has no on‑page topical signal for the majority of templates.

Pages with no H1 include: /events, /about-us, /faq, both audited blog posts, /chef/francesco-nyc, /filter (the main browse experience, 84 images, 2,281 words, no headings at all), and /contact-us.

Worse: /blog/how-much-is-a-personal-chef-less-than-you-think ships a <title> of literally "| Chefpost", <meta description> = null, og:title = null. This is a high‑commercial‑intent page (people researching whether to hire) and it's invisible to search engines, social shares and LLMs alike.

Fix: Add real H1s to every page template (see page‑by‑page review, Section 05). Rewrite every blog post's title, meta description, og:title and og:description (they're all null or broken). Fix the og:image = "[object Object]" JS serialization bug on 7 pages. Eta: 1–2 dev days + 2 days copy.
5
CRITICAL · CORE WEB VITALS

Mobile LCP is 6.1 seconds. The site can't rank competitively until this is fixed.

Impact: Lighthouse mobile Performance = 0.65. Largest Contentful Paint fails Core Web Vitals.

DataForSEO Lighthouse on chefpost.com/ (mobile): Performance 0.65, Best Practices 0.69, A11y 0.88, SEO 1.00. The killer is LCP = 6.1s (Google's "Good" threshold is <2.5s). 4 render‑blocking JS files + 3 render‑blocking stylesheets. /meal-prep is even slower at LCP 4.5s.

Mobile is already 63% of clicks, and Google has been running mobile‑first indexing for 4 years. The site can't out‑rank takeachef.com or weeklymealprep.com on competitive commercial terms with these Core Web Vitals scores.

Layered on top: a placeholder GA4 ID G-12345HHDSD is firing alongside the real ID (G-41JH1G30DC), wasting hits and polluting reports.

Fix: Defer + async the 4 blocking scripts; inline critical CSS for the hero; serve a real LCP image (hero photo) with fetchpriority="high" and width/height attributes; preconnect the analytics domains; drop the accessibility‑overlay widget (does not produce WCAG compliance and adds 200kb of JS). Remove the placeholder GA4 ID. Eta: 3–5 dev days. Target: LCP < 2.5s, Performance > 0.85.
Bottom line: All five of these issues are fixable inside 30 days with a typical 2‑person dev cell. None require new content. None require new budget for tools or vendors. The compound effect is the difference between a site that grew from position 33 → 22 in 16 months and a site that can grow to a position 10 baseline (and start earning clicks rather than impressions).
